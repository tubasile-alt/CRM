# FASE 1 APROVADA — REESTRUTURAÇÃO DO SISTEMA DE CÓDIGOS DE PACIENTES

## OBJETIVO PRINCIPAL

O objetivo desta fase NÃO é corrigir o histórico.

O objetivo é criar uma nova política segura e definitiva para os códigos dos pacientes daqui para frente, preservando integralmente todos os dados já existentes.

O código do paciente passa a ser o identificador operacional principal do CRM.

Cada paciente deve possuir:

* Nome
* Código único
* Ficha única
* Histórico único
* Prontuário único

O código deve ser:

* Único
* Permanente
* Sequencial
* Não reutilizável
* Imutável após criação

---

# REGRA MAIS IMPORTANTE

## NÃO ALTERAR PACIENTES ANTIGOS

Nesta fase é proibido:

* Renumerar códigos antigos
* Corrigir códigos antigos
* Corrigir duplicados históricos
* Mesclar pacientes
* Excluir pacientes
* Alterar prontuários
* Alterar consultas
* Alterar procedimentos
* Alterar evoluções
* Alterar vínculos existentes

Todos os pacientes históricos devem permanecer exatamente como estão.

Exemplos:

```text

264 → permanece 264
596 → permanece 596
845 → permanece 845
```

Inclusive o código duplicado 264 identificado anteriormente NÃO deve ser corrigido nesta fase.

## REGRA DE TRANSIÇÃO

Verificar o maior patient_code já existente.

Se NÃO existir nenhum código >= 1001:

```text
Próximo código = 1001
```

Se já existir algum código >= 1001:

```text
Próximo código = MAIOR_CODIGO + 1
```

Exemplo:

```text
Maior código encontrado = 1038

Próximo paciente = 1039
```

# PROTEÇÃO CONTRA DUPLICIDADE DE CÓDIGOS

Atualmente foi identificado que existe risco de concorrência.

Duas pessoas podem criar pacientes simultaneamente e receber o mesmo código.

Isso precisa ser corrigido.

Implementar:

## UNIQUE CONSTRAINT

Adicionar:

```sql
UNIQUE(doctor_id, patient_code)
```

Objetivo:

Impedir definitivamente que dois pacientes tenham o mesmo código para o mesmo médico.

---

## GERAÇÃO TRANSACIONAL SEGURA

Substituir a lógica simples:

```python
max_code + 1
```

por geração protegida por transação.

Objetivo:

Impedir race conditions.

Garantir que dois usuários nunca recebam o mesmo código.

---

# PROTEÇÃO CONTRA DUPLICIDADE DE PACIENTES

## NÃO CORRIGIR DUPLICADOS HISTÓRICOS

Nesta fase:

* Não mesclar
* Não excluir
* Não corrigir

Apenas prevenir novos problemas.

---

## NOVA VALIDAÇÃO

Antes de criar paciente:

Normalizar o nome:

* remover espaços extras
* remover espaços finais
* ignorar maiúsculas/minúsculas

Exemplos:

```text
FELIPE RUSSO
Felipe Russo
FELIPE  RUSSO
```

Devem ser considerados equivalentes.

---

## ALERTA DE POSSÍVEL DUPLICIDADE

Se encontrar paciente com nome igual ou muito parecido:

Exibir:

```text
Possível paciente já cadastrado.

Deseja abrir a ficha existente
ou criar um novo cadastro mesmo assim?
```

Opções:

* Abrir ficha existente
* Criar novo paciente mesmo assim

O sistema não deve decidir sozinho.
